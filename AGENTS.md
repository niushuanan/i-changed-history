# Prototype Instructions

Run the local server yourself and open the preview in the browser available to this environment. Do not give the user server-start instructions when you can run it.

Before making substantial visual changes, use the Product Design plugin's `get-context` skill when the visual source is unclear or no longer matches the current goal. When the user gives durable prototype-specific design feedback, preferences, or decisions, record them in `AGENTS.md`.

When implementing from a selected generated mock, treat that image as the source of truth for layout, component anatomy, density, spacing, color, typography, visible content, and hierarchy.

## Durable Product Decisions

- The product title is exactly `哎！我改变了历史？`.
- The player is a modern Chinese traveler, not an abstract observer. The first screen is the complete chronological filmstrip of fifty historical moments; there is no profile, personality test, MBTI label, or recommended subset before play.
- Opening cards must be famous, concrete AD historical turning points with an exact date, place, assigned role, immediate decision, deadline, and actual outcome.
- The deck must include substantial Chinese history before 1840 and frame world history in language accessible to Chinese players.
- The fifty-card deck is fixed at exactly 30 pre-1840 Chinese moments and 20 world moments. World cards must use people or events broadly recognizable to ordinary Chinese players; do not spend slots on minor-country or specialist Western history.
- Public-memory fiction may inspire a counterfactual hook, but the card itself must label and anchor real history. Preserve dramatic choices without turning sexual violence into a playable entertainment mechanic.
- All 50 history moments are always visible in one chronological horizontal filmstrip. Never reintroduce a recommended subset, shuffle, expand-all action, archive modal, or second browsing layout.
- The run has one authoritative timeline containing only the player's decisions. Never generate a hidden personality timeline or let a personality system choose on the player's behalf.
- During play the user may use direct rewrites without a per-run quota to declare completed historical results. Each rewrite still consumes the current node's single decision. The declaration is player-authored canon: the client constructs the immediate canonical receipt and trusts no model-authored outcome text; DeepSeek classifies impact and performs the creative world simulation in the following node.
- Preserve the current coal, newsprint, vermilion, teal, and yellow visual system unless the user explicitly requests a new visual direction.
- A complete run has exactly 12 playable decisions within one protagonist's biological lifetime. The alternate 2026 report is a separate posthumous epilogue and never consumes one of the twelve nodes.
- The protagonist keeps one fixed name and body across all twelve nodes, ages monotonically, may change role, faction, city, and social reach, and dies after the twelfth decision. Never use consciousness relay, reincarnation, descendants, or a replacement identity as the playable character.
- Early nodes provide tight feedback on the scale of the same day, three days, and six weeks. Later nodes spread across the same person's career and late life; modern starts compress the lifespan so the death still occurs before the 2026 report.
- After node 3, the opening event cannot remain the current plot, title, objective, or role. The same protagonist must enter a different consequential conflict caused by earlier choices rather than randomly changing topic.
- Surprise comes from changing the social carrier, conflict, institution, or affected population, not from forced travel. A timeline may remain in China; prefer real people, institutions, cities, and lived experiences familiar to Chinese players, while changing at least two narrative dimensions from the recent three nodes.
- Active gameplay always provides an exit control that clears the run and returns to the fifty-moment filmstrip.
- Event screens target the iPhone 13 viewport at 390 x 844 and must show only the scene, complete event copy, causal proof, all three generated choices, and the compact fourth-path entry without page scrolling. Do not restore the four-metric dashboard or repeated mission explanations.
- Scene art must follow the simulated year as well as the chapter. Never show modern or space technology in a pre-industrial node merely because it is a late chapter.
- A refresh or development-server reconnect during generation must resume the saved AI request automatically. Harmless non-narrative drift may be normalized locally, but visible historical prose is always AI-authored. A parseable response repairs only its invalid top-level AI fields and merges them back into the untouched scene; field-level validation must report missing fields and invalid narrative fields together so one compact patch can fix both. Unparseable or cross-field semantic failures receive at most one compact full repair. If repair fails, preserve the run and show an explicit retry instead of inventing a generic local historical scene.
- DeepSeek turn and ending requests use an 8192-token output ceiling so transport truncation cannot strand normal JSON; concise output is controlled by the prompt and schema, not a small token cap.
- Loading-room motion includes archival drift, causal-thread tracing, pulse nodes, exposure passes, and status cycling, with a `prefers-reduced-motion` static fallback.
- Loading progress is monotonic and honest: connected, reasoning, writing, validating, and repairing states come from the actual streamed request rather than timers. The steps and track never loop back or pretend to reach 100%. Use `新历史正在成形` rather than `因果推演` as the player-facing frame.
- Every playable event is model-generated and must show `DeepSeek 实时生成`. Local deterministic code may preserve a player-authored rewrite receipt, but it must never invent a playable historical scene or a 2026 report.
- All fifty moments live in the same chronological horizontal carousel. Do not reintroduce recommendations, expansion states, a modal, or a separate archive layout.
- Every history seed owns a local `/assets/history/<seed-id>.webp` path with an event-accurate source or a local period-art fallback; broken remote images must never be part of runtime rendering.
- The picker timeline and card filmstrip are one synchronized control: scrolling cards centers the matching year, and selecting a year scrolls the matching card into view.
- History picker cards must leave a clearly visible preview of the next card on the right at first glance. Keep the parenthetical swipe hint `（滑动可切换不同的历史瞬间）` visible beside the year/count readout.
- Active play is a continuous cinematic surface, not a dashboard. Use one scene, one compact causal receipt, and three separated action rows; avoid nested cards, metrics, badges, and explanatory panels.
- The AI is framed as a grounded world simulator. It generates the three ordinary actions against historical constraints, but it never vetoes a direct player rewrite. For direct rewrites, the client owns the immediate receipt and the AI owns only the next-node world extrapolation. The visible proof is source provenance, a causal receipt, and the same protagonist's age and evolving role, not long model prose.
- Personality is not part of the product. Do not restore profile forms, MBTI dimensions, personality-tailored choices, causal-preview personality lenses, or personality comparison reports.
- Music begins on the first eligible user gesture in the filmstrip and continues through generation, events, and ending. Browser autoplay failure must never block play.
- A direct rewrite enters the timeline only after a valid canonical result is committed. Network failure, model repair, refresh recovery, and retry never record a rewrite; player-authored results enter the same causal memory and 2026 ending contract as generated choices.
- Every one of the 12 playable nodes must be a consequential alternate-history turning point. Do not use ordinary reporting, routine administration, or generic livelihood scenes merely to create variety.
- The twelfth node is the protagonist's final playable decision, not a death cutscene. After that decision becomes canon, the ending must show the protagonist's death, four posthumous historical eras, three concrete 2026 life details, and a novel-like closing passage about a world the protagonist never saw.
- Every completed action enters a deterministic client-owned world canon before the next model call. Player-authored rewrites are irreversible facts, retain their exact wording, and remain explicit active mandates for the next three nodes; multiple mandates such as accession plus industrialization must coexist rather than collapse into one label.
- The client must never choose the next conflict from a fixed list of plot categories. DeepSeek receives the complete immutable canon, recent scenes, protagonist age, and historical constraints, then performs open-ended first-, second-, and third-order inference to choose the next consequential conflict.
- DeepSeek is the scene director, while the client owns continuity. Every turn must include at least two era-accurate historical anchors and three executable actions with actor, action, target, and deadline. Generic actions are rejected for model repair; failure remains retryable and never becomes a generic local scene.
- Turn and ending calls use DeepSeek high-reasoning mode, an 8192-token ceiling, and a 90-second transport window. A normal node uses one compact coherent turn request; never split one scene into blind parallel copy fragments. The ending uses two concurrent requests with disjoint ownership: biography and posthumous 2026 world report.
- Opening and continuation calls share one stable static system-protocol prefix so DeepSeek context caching can reuse invariant rules; all changing canon and chapter data stays in the final user message. Turn responses stream over SSE, but token fragments never enter persisted game state.
- Continuation context is tiered rather than raw chat history: send every prior decision as a compact life index, the latest decision in full, the latest three consequences in detail, every player-authored rewrite verbatim, the latest persistent ledger, and the latest three scenes only for anti-repetition. Never resend all prior scene prose.
- The biography ending receives all twelve years, ages, roles, decisions, direct results and costs. The 2026 report receives the complete decision canon and final persistent world ledger. The client merges both independently validated reports into one ending.
- Event screens place the three actions before a compact bottom `历史对照`. It must stack `你的时间线`, `正史原本`, and `为何改变` as three full-width rows. Each conclusion is scan copy, not an essay: alternate state and causal bridge are at most 36 Chinese characters, the factual baseline is at most 48, and the prior choice is never repeated here. The real-history field never repeats the UI label or mixes in the alternate timeline; never restore the square `因果回执` block or repeated `你的决定 / 已经改变 / 重大节点` labels.
- Every newly generated event `narrative` contains 96-160 Chinese characters in exactly three complete sentences. Sentence one establishes the famous-event prehistory or how the previous decision produced the current situation; sentence two names the real people, institution or faction, what they want, and visible evidence; sentence three states the protagonist's usable leverage, immediate decision, and direct cost of failure. Never line-clamp, ellipsize, or silently hide AI-authored event prose. Loading an older save may preserve its shorter original narrative verbatim.
- Event pages use an adaptive vertical height system at 390 x 844: the progress header occupies normal flow above the scene image, never overlays it; copy and causal proof own only their natural height; rich turns reduce type and use a 300px scene, while a rich opening may use 330px because its factual anchor is shorter than the continuation comparison; the three actions receive the remaining viewport without clipping or a blank lower half; the fourth-path button keeps an 8px bottom safe area.
- Scene captions always use two full-width rows: the complete year/life-stage/age string on the first row and the location on the second. Never return to a left-right caption that lets a long location squeeze or split the time.
- The ending has exactly two report pages for the same player timeline. Page one is a `史记`-style biography spanning all twelve decisions with both modern Mandarin and classical Chinese. Page two is a one-screen novel-like butterfly-effect account from the protagonist's death to ordinary life in 2026. Neither page may mention personality.
- The public repository must install and build from an arbitrary user directory on Windows and Linux. Runtime files, package configuration, and scripts must never contain a developer-specific absolute home path; keep `npm run check:portability` and the cross-platform GitHub Actions build green.
- Historical locations must use vocabulary appropriate to their year and culture. For pre-1900 scenes, reject modern generic space names such as `议事厅`, `会议室`, `办公室`, and `指挥中心`; repair the AI-authored `location` field instead of locally rewriting visible prose.
- The immediate post-decision screen is a plain result confirmation, not a system-analysis report. Use the concrete landed result as the heading, followed only by the player's choice, its cost, beneficiary, and payer. Never show `因果回响`, `世界已回应`, per-step deviation points, or low-contrast microcopy on this screen.
