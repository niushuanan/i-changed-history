# 现场扫码版生成性能规格

## 目标

在不减少十二次决定、不改用本地假剧情、不削弱玩家自由改写权威的前提下，让现场扫码玩家更快看到并操作下一幕，同时把结构修复和长局中断降到可比赛演示的水平。

## 基线

基于 `tmp/soak/final-10-20260714/` 的十局真实 DeepSeek 长测：

- 十局全部完成，平均 8.36 分钟/局。
- 110 次续幕平均 37.81 秒，P50 36.51 秒，P90 53.24 秒。
- 45 次自由改写裁决平均 7.96 秒，之后仍需再请求下一幕。
- 45 次首答需要结构修复，其中续幕 34/110，首答修复率约 31%。
- 结局双报告平均 49.88 秒，P50 45.82 秒。

## 产品原则

1. 玩家写下的是已经发生的结果，不是交给模型审批的行动申请。提交后同步写入客户端正史，模型只能推演后果，不能否定、降级或改写成败。
2. 每幕仍由 DeepSeek 完整生成。客户端只拥有身份、时间、玩家原文、因果账本和结构校验，不生成可玩历史现场或 2026 报告。
3. 快路径必须经过与高推理路径相同的结构、历史锚点、人物连续性、具体行动和正史矛盾校验。快路径不合格时才升级为高推理恢复。
4. 流式草稿只用于让玩家提前阅读，不进入存档，也不能提前点击；完整响应通过校验后才成为当前幕。
5. 不把一个幕次拆成彼此不知道上下文的并发文案请求，不预生成 A/B/C 三条分支，不用额外请求换取表面速度。

## 方案

### 1. 删除自由改写的重复裁决

- 自由改写提交时由客户端同步构造 `玩家钦定` 回执并写入 `PlayedTurn`。
- 默认按 `rupture` 计算单幕影响，因为自由改写的产品语义就是直接声明已经改变的结果；该数值只影响本地展示，不决定事实是否成立。
- 同一次状态转移直接启动下一幕或双报告，不再调用 `adjudicateCustomAction`。
- 原文、传播媒介和直接结果永久进入 `lifeIndex`、`playerCanon`、`WorldCanon` 与结局输入。

### 2. 快路径与高推理恢复

- 普通续幕先使用 V4 Flash 非思考模式，保留完整提示、8192 上限、SSE 与同一校验器。
- 首答结构或语义不合格时，不做同等成本的多轮高思考字段补丁：小范围字段补丁走非思考模式；补丁仍失败时只允许一次高推理完整恢复。
- 双报告继续高推理并发生成，避免牺牲最终作品质量。
- 所有模式、首字时间、完成时间、缓存命中、token 与修复阶段进入脱敏指标。

### 3. 缩小输出合同

- 删除没有运行时消费者的 `rippleLens` 字段和对应提示、类型与测试。
- 只让模型输出 AI 真正拥有的字段；客户端权威字段继续在解析时注入。
- 把玩家最先需要阅读的 `headline`、`narrative`、现场目标、倒计时和 choices 排在 JSON 前部，便于流式草稿尽早成形。
- 保留两个以上真实时代锚点、三条具体行动、丰富前情与真实历史对照，不以删文案换速度。

### 4. 流式可读草稿

- 传输层从 SSE 内容增量解析已经闭合的 JSON 字符串字段。
- 当标题和完整前情出现时，等待页切换为“现场正在写成”的可读草稿；选项未完整校验前不允许操作。
- 草稿更新限频，刷新或断线时仍依赖已有持久化请求恢复机制。

### 5. 正史强化

- 当前幕展示仍在生效的玩家原文，最多展示最近一条，避免增加面板。
- 下一幕提示要求 `worldStateChange` 和 `causalBridge` 明确兑现最近自由改写；权威因果账本逐字注入最近三幕。
- 长测检查每条自由改写逐字进入正史、后三幕账本和最终人物年表，并检查后果中不存在否定语义。

## 验收目标

以同一模型、同一网络、十个不同历史开局、每局十二节点且四至五次不同自由改写为准：

- 完整通关率至少 9/10，目标 10/10；本地可玩场景/结局兜底为 0。
- 完整一局 P50 相对 8.36 分钟基线至少降低 35%，即不高于 5.43 分钟。
- 续幕可操作完成 P50 不高于 22 秒，P90 不高于 35 秒。
- 首个可读场景 P50 不高于 8 秒，P90 不高于 15 秒。
- 自由改写不再产生单独 API 调用，提交到下一幕的等待与普通选择一致。
- 续幕首答修复率不高于 10%；单幕最多一次字段补丁和一次高推理恢复。
- 每条玩家原文 100% 逐字保存在正史和人物年表；活跃后三幕 100% 出现在权威账本；矛盾接受率为 0。
- 普通单测、类型检查、生产构建、跨平台检查和 390 x 844 实际浏览器流程全部通过。

## 不做

- 不在公开前端中加入模拟 AI 或固定后续剧情。
- 不并发生成三条未来分支。
- 不自建 GPU 推理服务。
- 本轮不把 API 密钥迁移到服务端；但正式公开扫码前必须用服务端/边缘代理保护密钥并做限流。

## 调研取舍

- [DeepSeek Context Caching](https://api-docs.deepseek.com/guides/kv_cache/)：缓存按完全一致的输入前缀命中，因此把稳定系统协议放在前面、变化中的世界正史只放最后一条 user message，并采集 hit/miss token。
- [DeepSeek Rate Limit & Isolation](https://api-docs.deepseek.com/quick_start/rate_limit/)：现场并发应由服务端按匿名 `user_id` 隔离并处理 429；本轮纯前端先实现有界 `Retry-After`、指数退避和抖动，公开部署仍需边缘代理。
- [Vercel AI SDK](https://github.com/vercel/ai)：借鉴流式结构化 UI、usage/telemetry 与统一 provider 的设计，但本项目保留轻量原生 SSE，避免为单供应商原型引入大框架。
- [BAML](https://github.com/BoundaryML/baml) 与 [Instructor](https://github.com/567-labs/instructor)：吸收“schema 是一等合同、校验失败要带证据修复”的思路；当前 TypeScript/Zod 链路已能做根字段补丁，无需迁移 DSL/Python。
- [LiteLLM](https://github.com/BerriAI/litellm)：借鉴网关层重试、限流、观测与供应商切换；这部分列为公开扫码前的边缘代理工作，不在浏览器内伪造多模型容灾。
- [LMCache](https://github.com/LMCache/LMCache)：验证 KV cache 对稳定前缀的重要性；本轮使用 DeepSeek 托管缓存，不自建 GPU/KV 基础设施。

## 最终验收

真实生产链路日志位于被 Git 忽略的 `tmp/soak/release-acceptance-final-10-20260714/`：

- 10/10 个不同历史开局完成 12 次决定和双报告，45/45 条唯一自由改写进入正史，0 次整幕人工重试，0 个本地可玩场景或结局兜底。
- 单局 P50 153.81 秒，相对 8.36 分钟基线降低 69.3%，通过 326 秒目标。
- 续幕 P50 10.55 秒、P90 14.98 秒，通过 22/35 秒目标。
- 首段可读 P50 3.15 秒、P90 4.16 秒，通过 8/15 秒目标。
- 自由改写本地同步提交 P50/P90 均为 0 毫秒，不产生裁决 API 请求。
- 110 个续幕中 7 个需要首答字段修复，修复率 6.36%，通过 10% 目标；无高推理续幕恢复。
- DeepSeek 提示缓存命中 token 比例 46.8%。

后续对“开场事件可作为因果源但不可继续垄断主线”的校验做了精确收敛，并在 `tmp/soak/causal-source-verification-2-20260714/` 补跑赤壁与玄武门：2/2 通关、22 个续幕修复率 0。
