# 快速稳定推演与自适应事件页设计

## 问题证据

- 当前首幕要求 DeepSeek 一次返回约 30 个字段，包含客户端已知道的章节、年龄、年份、上次回响和四个未展示指标。
- 实测罗马大火首幕：第一次请求 13.934 秒，整体叙事有效，仅因两个选项同时标记 `usesModernKnowledge` 而失败；随后完整修复请求再耗时 12.049 秒，总等待约 26 秒。
- 续幕 prompt 同时发送完整 `playedHistory` 和重叠的 `WorldCanon`；轮次越后，上下文越长。
- 事件页用固定的文案高度和因果高度接住可变文字：长文会被裁切或溢出遮挡，短文则留下空洞。

## 方案决策

不将一幕盲目拆成三个并发模型请求。正文、因果与选项共享同一个“当前重大冲突”，独立生成会产生三个不同现场，合并后反而不可玩。

采用混合架构：

1. 每个可玩节点由一次紧凑的“幕次导演”请求生成，保证现场、因果和三个行动一致。
2. 章节、章节名、主角年龄、人生阶段、权威年份、上次回响、指标和生成来源改由客户端组装，不再让模型复述。
3. 结构归一化负责修正已知的无害漂移：超长文案裁成完整短句，A/B/C 和偏离等级按位置权威化，三项中只保留一项现代知识标记。只有缺失核心叙事时才发起一次精简修复，不再第三次整页重生。
4. 结局拆为两个真正独立的并发请求：`穿越者列传` 只生成白话、文言、死亡和十二决；`被改变的 2026` 只生成四段身后历史、普通生活与尾声。两份数据分别校验后由客户端确定性合并。

## 第五轮及后续轮次的上下文

模型不只看上一轮，也不接收前四轮的全部原始正文。客户端编译一份 `NarrativeContext`：

- `lifeIndex`：截至当前的全部决定，每幕只保留章节、年份、身份、决定原文和直接结果。
- `latestDecision`：上一次决定的完整回响，作为下一幕的近因。
- `activeConsequences`：最近三幕的结果、代价、受益者和承担者。
- `playerCanon`：所有玩家直接改写的原文、传播机制和生效章节，永不压缩或改写。
- `persistentLedger`：上一幕仍在生效的因果账本。
- `recentScenes`：最近三幕的标题、地点、身份和时代锚点，只用于防止重复。

因此第五轮能看到前四轮的决策链，但不会因重复叙事而浪费 token。

## 结局上下文

- 人物列传需要全部十二幕的年份、年龄、职位、决定原文、直接后果与代价，但不需要每幕的现场正文。
- 2026 世界报告需要全部不可撤销决定、所有玩家钦定、十二幕直接后果和最终因果账本，但不需要身份过渡或现场物件。
- 两个请求共用同一份客户端权威决定索引，不共享模型生成文字，因而可安全并发。

## 事件页布局

- 页头与图片始终在正常网格流中，禁止文字绝对定位覆盖正文。
- 场景图高度按文案密度在 132/146 像素两档切换；长文为操作区让出空间。
- 正文与因果回执使用内容高度，不设固定 `max-height`；所有可见 AI 字段在 schema 层缩短为完整短句。
- 决策区占据所有剩余高度，三个行动均分；短文时增大可点区，而不留白。
- 390 x 844 无页面滚动、无文字相互覆盖，三个行动和直接改写始终可见。

## 失败处理和验收

- 正常幕次只发起一次 DeepSeek 请求；已知可归一化漂移不得触发第二次。
- 核心字段缺失时最多一次精简修复；修复只带错误列表、原输出和权威字段，不重发全部历史。
- 结局测试必须证明两个请求在第一个完成前都已启动，且合并后仍通过完整 `AlternatePresent` schema。
- 真实 DeepSeek 首幕请求记录耗时与请求次数；目标是将本次基线的两次请求降为一次。
- 在 390 x 844 实际浏览器中检查开场幕和续幕两种页面，比较 `scrollHeight`/`clientHeight`，并人工查看截图。
