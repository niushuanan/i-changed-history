# 十二节点长局稳定性设计

## 问题与根因

当前中断不是单纯的网络波动。续幕协议同时要求：

- `causalLedger` 最多三项；
- 每一条玩家自定义正史都必须在此后所有幕次逐字进入 `causalLedger`；
- 解析器对全部历史自定义正史逐项检查，不区分它是否仍处于三幕强制兑现期。

因此一局出现四到五次自定义结果后，模型面对的是不可同时满足的合同。节点越晚，累计决定、场景防重复、主角连续性和严格字段约束越多，结构修复也更容易再次失败。当前错误页只保留 `invalid_structure`，没有暴露首轮和修复轮究竟失败在哪个字段，无法形成可迭代的可靠性数据。

## 方案比较

### 方案 A：直接放松所有 schema

可以提高表面通过率，但会允许缺字段、抽象选项和人物断裂进入游戏。它牺牲生成质量，不采用。

### 方案 B：失败后盲目多次整幕重试

可能提高偶然成功率，但会显著增加等待与 API 成本；如果提示词自身矛盾，重试不会解决。它只能作为最后一道有上限的恢复，不作为主方案。

### 方案 C：分层正史注入与有证据的有限恢复

客户端继续保存全部十二次不可撤销决定；每条玩家自定义结果只在后续三幕作为“活跃正史”强制逐字进入因果账本，三幕以后仍保留在完整人生索引和结局上下文中，但不再占用当前幕最多三项的账本。解析时由客户端注入活跃正史，模型无需完成机械抄写，只负责把它推演为当前冲突。该方案同时消除逻辑冲突、保留玩家主权并降低模型结构负担，采用此方案。

## 新的上下文合同

每个续幕请求包含四层数据：

1. `lifeIndex`：截至当前的全部决定与直接结果，作为不可撤销人生正史。
2. `activePlayerCanon`：只包含当前章节之前三幕内的玩家自定义结果，最多三项，逐字保留，并要求本幕可见后果继续兑现。
3. `latestDecision` 与 `activeConsequences`：上一决定完整回响和最近三幕具体代价，提供近因。
4. `recentScenes`：最近三幕的标题、身份、地点与历史锚点，只负责防止同一事件线机械重复。

模型仍开放决定下一处重大历史冲突。客户端不选择剧情类别、不补写标题、正文、选项或 2026 报告。客户端只拥有章节、年龄、年份、上一回响、活跃正史账本和其他确定性连续性字段。

## 校验与恢复

- 首轮输出先经过无害归一化、Zod 字段校验和人物连续性校验。
- 可解析且只有根字段失败时，只请求这些字段的补丁并合并回原稿。
- 补丁仍失败或出现跨字段语义错误时，允许一次带精确错误清单的紧凑恢复生成。
- 三轮仍失败才进入错误页；绝不生成本地可玩场景或伪造结局。
- 每次失败只记录目标、阶段、字段路径和错误摘要，不记录 API Key，也不把完整模型正文写入版本库。
- 自定义结果裁决即使降级为客户端正史回执，后续可玩幕次仍必须由 DeepSeek 生成；压力测试单独记录这种降级次数。

人物列传不再要求在 `consequence` 中机械重复玩家原句。客户端已经权威写入 `historyTimeline.playerChoice`；模型只需生成不否定该事实的后果，减少四到五条长自定义结果对结局 JSON 的重复负担。

## 真实十局压力测试

建立显式运行的真实 DeepSeek soak 工具，不进入普通单元测试：

- 选择十个不同历史入口，覆盖中国与世界、古代与近现代；
- 每局完成十二个可玩节点并生成两份结局；
- 每局在预定的四到五个不同章节提交不同的玩家自定义结果；
- 全部自定义文本在本次十局中唯一；
- 普通节点轮换 A/B/C，避免只验证单一路径；
- 记录每幕耗时、主请求失败、字段修复、恢复生成、自定义裁决降级、终止章节和结局状态；
- 中间探针失败用于继续修改，不计入最终十局；最终批次必须从空状态重新运行。

## 验收标准

- 最终十局至少九局到达双报告结局，完整通关率不低于 90%。
- 每局恰好十二次决定，其中四到五次为不同的玩家自定义结果。
- 所有成功续幕均为 `generationSource: "deepseek"`，不存在本地可玩场景兜底。
- 每个成功幕次的章节、主角姓名、年龄和人生阶段连续，活跃玩家正史逐字存在于客户端权威账本。
- 失败能够归类到传输、首轮结构、字段修复、语义恢复或结局，而不是只得到笼统的 `invalid_structure`。
- 普通 Vitest、TypeScript、生产构建、可移植性扫描和真实 390 x 844 浏览器流程继续通过。
