# 流式高质量历史推演设计

## 产品目标

在不拆散一幕叙事、不伪造 AI 内容的前提下，同时满足三件事：

1. **加速**：利用回响页预取、DeepSeek SSE、上下文缓存和小字段修复缩短可感知等待。
2. **真实 AI 驱动**：玩家看到的现场、因果和行动都必须来自 DeepSeek；客户端只负责权威事实、校验、合并与展示。
3. **生成质量**：继续用一个模型请求统一导演现场、因果和三个行动；结构漂移只修坏字段，不重写已经正确的历史内容。

## GitHub 调研吸收

- `vercel/ai`：吸收流式生成的状态机与“完整结构校验后再提交 UI”原则，不把未闭合 JSON 直接展示给玩家。
- `BoundaryML/baml`：吸收 schema-aligned parsing 思路，把模型的局部结构漂移视为可修复数据，而不是整页报废。
- `567-labs/instructor-js`：吸收 Zod 校验、带错误反馈的重试与增量输出思想；修复请求只携带失败字段和权威上下文。
- `langchain-ai/langgraph`：吸收 durable execution/checkpoint 思路；继续把请求意图和已完成结果持久化，刷新后恢复而不是重演已完成节点。

不直接引入这些框架。当前产品是纯前端且已经有 Zod、reducer、localStorage 和 DeepSeek 传输层，复制核心模式比增加运行时依赖更稳定。

## 为什么不拆成三个并发幕次接口

事件正文、因果证明和三个行动共享同一个冲突。如果三个模型分别生成，它们可能给出不同地点、不同权力关系和不同历史前提，合并后虽更快返回，却更不可信。并发只用于彼此独立的结局两份报告；可玩幕次始终由一次主推演完成。

## 新生成链路

1. 玩家作出决定时，客户端立即把决定写入世界正史，并在回响页后台开始下一幕请求。
2. DeepSeek 使用 `stream: true` 返回 SSE。传输层只上报四个可信阶段：连接、推理、成稿、完成；不展示未完成 JSON。
3. 主结果完成后通过 Zod 和因果约束校验。
4. 若仅部分根字段失败，第二次请求只返回这些根字段；客户端把 AI 修复字段合并回原始 AI 结果，再重新校验。
5. 若错误属于跨字段语义冲突或原始结果不可解析，才允许一次完整紧凑修复；仍失败则进入可重试错误页，不生成本地剧情。

## 缓存与上下文

- 所有幕次共享固定 system prompt 和固定 turn protocol，动态历史上下文放在最后一个 user message。
- DeepSeek 的前缀缓存可命中稳定的规则与结构协议，减少后续幕次首 token 延迟。
- 第五轮及以后仍使用 `NarrativeContext`：全量决定索引、最近三幕后果、全部玩家钦定原文、最新账本和最近场景；不发送全部旧正文。
- 结局继续并发生成列传与 2026 世界报告，各自只读取需要的十二幕压缩上下文。

## 真实 AI 边界

- 客户端可以注入章节、年龄、年份、上一回响和指标，因为这些是规则，不是故事。
- 客户端不得补写标题、叙事、因果证明或选项文案；缺失时必须由 DeepSeek 字段修复。
- 玩家自由改写的结果由玩家本人拥有，客户端只将原文确认为正史；DeepSeek负责推演它的传播和后果。
- 页面只在完整结构通过校验后显示 `DeepSeek 实时生成`。

## 验收标准

- 正常幕次一条主请求；局部字段失败时第二条请求只包含失败根字段。
- SSE 即使跨任意字节块切分，也能无损拼接中文 JSON，识别 `[DONE]` 和 usage。
- 等待页阶段来自真实网络事件，不再靠固定 1.05/2.25 秒假进度。
- 请求中存在稳定、完全相同的 turn protocol 前缀，动态上下文位于最后。
- 全套测试、类型检查、构建通过；390 x 844 浏览器实测可进入历史选择并完成真实 DeepSeek 首幕。
